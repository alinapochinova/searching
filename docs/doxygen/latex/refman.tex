\chapter{Результаты экспериментов}

\section{Сравнение всех методов поиска}
\begin{center}
\includegraphics[width=0.9\textwidth]{search_comparison.png}
\end{center}

\textbf{Анализ результатов:}

\begin{itemize}
    \item \textbf{Линейный поиск (красный):} Время растёт линейно O(n). 
          При N = 1 000 000 достигает 32513 мкс (32.5 мс) на один поиск.
          Непригоден для больших массивов.
    
    \item \textbf{BST (синий):} Время почти не зависит от N (4 мкс при N=1 млн),
          так как дерево содержит только 2000 уникальных ключей.
          Дубликаты хранятся в векторах, не создавая новые узлы.
    
    \item \textbf{RBTree (зелёный):} Результаты аналогичны BST (3.4 мкс),
          но немного медленнее из-за хранения цвета и родителя.
          Преимущество RBT проявится на данных с большим числом уникальных ключей.
    
    \item \textbf{HashTable (фиолетовый):} Постоянное время O(1) (3.2 мкс),
          коллизий мало (475 при N=1 млн).
          Лучший выбор для поиска по точному совпадению.
    
    \item \textbf{std::multimap (оранжевый):} Хранит каждый дубликат отдельно,
          поэтому при N=1 млн содержит 1 млн узлов.
          Время поиска 11.1 мкс (высота дерева около 20 сравнений).
\end{itemize}

\textbf{Вывод:} Для данных с повторяющимися ключами группировка дубликатов в узле
(BST, RBT, HashTable) даёт значительный выигрыш по сравнению со стандартным multimap.
Линейный поиск следует использовать только для очень малых массивов (менее 1000 элементов).

\section{Зависимость времени поиска от числа уникальных ключей}
\begin{center}
\includegraphics[width=0.9\textwidth]{unique_keys_plot.png}
\end{center}

\textbf{Анализ:}
\begin{itemize}
    \item При фиксированном размере массива (1 млн) менялось количество уникальных ключей от 100 до 500 000.
    \item BST и RBT: время растёт логарифмически O(log U).
    \item HashTable: время остаётся константным (размер таблицы 200003).
    \item При U = 500 000 (50 процентов уникальных ключей) BST требует около 19 сравнений, HashTable 2-3.
\end{itemize}

\textbf{Вывод:} Чем больше уникальных ключей, тем больше преимущество хэш-таблицы.
При малом числе уникальных ключей (U менее 10 000) разница между структурами незначительна.

\section{Влияние размера хэш-таблицы на коллизии и время поиска}
\begin{center}
\includegraphics[width=0.9\textwidth]{hash_size_plot.png}
\end{center}

\textbf{Анализ:}
\begin{itemize}
    \item Фиксированный массив: 500 000 элементов, 2000 уникальных ключей.
    \item При малом размере таблицы (101) коэффициент загрузки около 20,
          много коллизий, время поиска высокое.
    \item Оптимальный размер: примерно 4000 (2 умножить на число уникальных ключей),
          коллизии минимальны, время стабильно.
    \item При дальнейшем увеличении размера (200 000 и более) коллизии стремятся к 475,
          время не улучшается, но память тратится впустую.
\end{itemize}

\textbf{Вывод:} Размер хэш-таблицы должен быть в 2-4 раза больше числа уникальных ключей.
Слишком маленькая таблица ведёт к многим коллизиям, слишком большая к избыточному расходу памяти.

\section{Общие выводы по лабораторной работе}

\begin{enumerate}
    \item \textbf{Линейный поиск} только для очень малых массивов (менее 1000 элементов).
    
    \item \textbf{BST и RBT} эффективны при группировке дубликатов.
          При малом числе уникальных ключей работают почти как хэш-таблица.
          RBT обеспечивает гарантированную логарифмическую высоту.
    
    \item \textbf{Хэш-таблица} лучший выбор для поиска по точному совпадению
          при большом количестве уникальных ключей (среднее O(1)).
          Размер таблицы должен быть в 2-4 раза больше числа уникальных ключей.
    
    \item \textbf{std::multimap} удобен, но уступает самодельным структурам с группировкой дубликатов.
    
    \item \textbf{Хэш-функция} критически важна для производительности.
          Для строк с номерами рейсов лучшей оказалась JS-функция.
    
    \item \textbf{Коллизии} растут с увеличением числа уникальных ключей при фиксированном размере таблицы.
          Оптимальный размер таблицы 2-4 умножить на число уникальных ключей.
\end{enumerate}
